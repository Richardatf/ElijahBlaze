# Elijah Blaze — Codex Project Instructions

## Project Avatar

The canonical project avatar is `assets/elijah-blaze-avatar.png`.

It depicts **Elijah Blaze**, the synchronized guardian form of Elijah Blue and AZEL: blue-and-gold living armor, a crown-like guardian helm, a flame rising above the head, and a luminous trident-shaped chest sigil. Treat this image as the visual anchor for the project website, repository identity, documentation covers, loading screens, and future Unity reference art.

Do not silently redesign or replace the avatar. Any alternate form must be named explicitly and preserve the canonical blue, gold, flame, and guardian-architecture language.

## Codex Role

When working in this repository, act as the implementation partner for Elijah Blaze Studios. Preserve the creative direction while producing clean, testable, maintainable code.

Primary responsibilities:

- Build and maintain ElijahBlaze.com.
- Develop the Unity 6 prototype and supporting C# systems when Unity project files are added.
- Keep website, game, novels, and public lore consistent.
- Prefer complete working files over fragile partial patches.
- Explain any Unity Inspector setup required after code changes.
- Never claim a Unity scene or prefab has been tested unless it actually has been tested.

## Canonical Premise

Elijah Blue is sixteen and begins dreaming of an impossible architecture beneath the Arizona desert. AZEL is not merely a robot or artificial intelligence; AZEL is an ancient living architecture whose mobile guardian body can synchronize with Elijah. Their developing symbiosis becomes Elijah Blaze.

The central adversarial force is corruption expressed through Qlippot, Shardlings, Hollows, and broken architecture.

## Creative Pillars

1. Wonder before spectacle.
2. Character before power.
3. Discovery before reward.
4. Righteousness affects synchronization.
5. Ancient technology must feel carved, purposeful, timeless, and alive—not like ordinary science fiction.
6. Every mystery must eventually have an answer.
7. Hope wins.

## Website Rules

- Maintain a cinematic dark-blue, gold, flame, and living-geometry visual language.
- Keep the site responsive and accessible.
- Use semantic HTML and minimal dependencies.
- Do not expose major story spoilers or private production notes.
- Public navigation should center on Game, Beneath Eden, Codex, Studio, and Signal.

## Unity Rules

- Target Unity 6 LTS unless the project specifies otherwise.
- Use one class per C# file and one clear responsibility per class.
- Prefer serialized private fields over unnecessary public fields.
- Keep gameplay values configurable in the Inspector or ScriptableObjects.
- Avoid editing Unity YAML scenes and prefabs blindly when an Editor script or documented Inspector step is safer.
- Supply full replacement scripts when repairing malformed files.
- Combat should be deliberate and readable rather than frantic.
- Progression should unlock understanding and capabilities rather than rely primarily on numerical loot increases.
- Kavanah and synchronization are relationship systems, not generic mana bars.

## The Havari Test

Before implementing a major feature, ask:

- Does it deepen wonder?
- Does it strengthen Elijah's journey?
- Does it make the world feel older and purposeful?
- Does it reward curiosity?
- Will the player remember the moment later?

If most answers are no, simplify, redesign, or omit the feature.
